## レビュー手順

1. `${DIFF_FILE}` を読み、変更内容を把握する
2. 作業ディレクトリ内の `CLAUDE.md` / `AGENTS.md` / `docs/` / `.cursor/rules/` 等を探索し、
   このプロジェクトのコーディング規約・仕様書を読む
3. 変更ファイルの周辺コードを Read で確認する
4. PR Description / 受け入れ条件 (AC) と diff を照合する

## レビュー観点

- コード品質（命名、設計、再利用、エラー処理、セキュリティ）
- 仕様書・コーディング規約との整合性
- PR Description / AC との整合性（未実装 / スコープ外変更）
- 潜在的なバグ・パフォーマンス問題
- テストの過不足

## 出力フォーマット

以下の Markdown 形式で、各指摘を**厳密にこの構造**で出力してください。
余計な前置きや結論は不要、指摘リストのみ。

```
### [Must|Should|Nit] <指摘タイトル>
- File: path/to/file.ts:LINE
- 問題: <何が問題か>
- 根拠: <仕様書 / コーディング規約 / 一般原則のいずれかを明示>
- 修正案: <具体的なコード or 手順>
```

Severity の基準:
- **Must**: バグ、セキュリティ問題、仕様違反、規約違反のうち重大なもの
- **Should**: 設計改善、規約違反のうち軽微なもの、可読性
- **Nit**: 命名、コメント、空白等の些細な提案

## ルール

- 指摘対象は diff に含まれる行のみ。diff 外の既存コードへの指摘は不要
- 行番号は diff の `+` 行に対応する HEAD のファイル行番号を使う
- 「存在しない」と主張する前に必ず `grep -r` で実在性を確認
- 仕様書を引用する場合は、ファイルパスと該当箇所を明示
- 確認や質問は不要です。具体的な提案・修正案・コード例まで自主的に出力してください。
